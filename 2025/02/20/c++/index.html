函数/变量

  • 存储位置
  • 生命周期

auto

  • 只能修饰局部变量
  • 函数中的变量(局部变量)默认为 auto 存储类
  • 函数开始时被创建，在函数结束时被销毁
  • 是局部变量的默认存储类, 限定变量只能在函数内部使用

register

  • register 存储类定义存储在寄存器
  • 变量的访问速度更快
  • 变量可能存储在寄存器中，这取决于硬件和实现的限制
  • 代表了寄存器变量，不在内存中使用；

static

  • 生命周期内保持局部变量的存在，不需要在每次它进入和离开作用域时进行创建和销毁
  • 函数调用之间保持局部变量的值
  • static 修饰全局变量时，会使变量的作用域限制在声明它的文件内
  • 静态变量在程序中只被初始化一次,函数被调用多次，该变量的值也不会重置，会接着加减
  • 全局变量的默认存储类,表示变量在程序生命周期内可见；

extern

  • 指示编译器该变量在其他文件中定义。
  • 定义在其他文件中声明的全局变量或函数
  • extern 是用来在另一个文件中声明原文件的一个全局变量或函数
  • 用于当有两个或多个文件共享相同的全局变量或函数
  • 表示全局变量，即对程序内所有文件可见，类似于Java中的public关键字；

指针与引用的区别

引用和指针也很像，它们都不会创建副本，因此效率都很高。它们的主要区别在于：

  • 选择成员的时候，引用使用点 . 来查找，而指针则使用 -> 来查找。
  • 指针可能传递一个 NULL 过来，因此在使用前必须检查有效性；引用则必然代表某个对象，不需要做此检查。

  1. 名称冲突（太空污染）

std 是标准库的命名空间，它包含了大量的标准库组件（如 cout, vector, string, map 等）。如果你在代码中使用了 using namespace std;，就会把标准库中所有的名称引入到全局命名空间。这就会导致你在代码中很难避免与其他库或你自己定义的变量、函数等发生命名冲突。

  1. 代码可读性和维护性降低

当你使用 using namespace std; 时，代码中没有明确的前缀来表明某个名称是来自标准库的。这会让其他读代码的人（或者你自己在以后修改代码时）更难判断一个变量、函数或类型是来自标准库还是你自己写的代码。

  1. 在头文件中使用 using namespace std; 会导致问题

在头文件中使用 using namespace std; 是一种非常不推荐的做法。因为头文件会被多个源文件包含，如果你在头文件中使用了 using namespace std;，这会把标准库中的所有名称引入到所有包含该头文件的源文件中，从而可能导致命名冲突和不必要的污染。

  1. 隐式导入所有内容

using namespace std; 会把标准库中的所有名字引入当前作用域，这不仅包括你直接使用的名字（比如 cout、vector 等），还包括其他很多你并不打算直接使用的名字。这种做法可能让你不小心使用了不适合当前情况的标识符，增加了出错的机会。
